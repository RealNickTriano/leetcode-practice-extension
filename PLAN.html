01
The Idea

Concept

You forget solutions on a curve. The extension schedules each problem to come back right before you'd forget it — so every repeat costs the minimum time for the maximum retention.

Today, picking what to practice on LeetCode is guesswork: you re-solve things you already know, or you grind new problems while old patterns quietly decay. Anki solved this exact problem for flashcards decades ago with spaced repetition: rate how hard a recall was, and the algorithm pushes the next review out on an expanding schedule.

LeetLoop applies that loop to coding problems:

  • Every problem you solve becomes a card with a review schedule.
  • When you visit leetcode.com/problemset/, the extension injects a "Due Today" card pinned above the problem table — the single most overdue problem in your deck.
  • When you get an Accepted submission on a due problem, you rate the attempt (Again / Hard / Good / Easy), and the scheduler computes the next review date.
  • No solves due? The extension suggests a new problem from a curated list (e.g. NeetCode 150 order), so the deck grows steadily.
Design principle One problem per day, surfaced where you already are. No separate app, no dashboard to remember to open. The habit lives on the page you already visit.

Spaced Repetition

The scheduling engine

A direct adaptation of Anki's SM-2: each card carries an ease factor and an interval; your rating after each solve updates both.

The rules

Each card stores ease (starts at 2.5), interval (days), and reps (consecutive successful reviews). After each rated solve:

RatingMeaningNext intervalEase change
Again Couldn't solve it, or needed the editorial 1 day — reps reset to 0 −0.20 (floor 1.3)
Hard Solved, but slow or with wrong turns interval × 1.2 −0.15
Good Solved at a normal pace 1 → 6 → interval × ease ±0
Easy Instant recall, clean solve interval × ease × 1.3 +0.15

Picking today's problem

Every page load, the scheduler ranks the deck and picks one:

  • Most overdue card first — sorted by now − dueDate, descending.
  • Tie-break by ease — weaker cards (lower ease) surface before stronger ones.
  • Nothing due? Introduce the next new problem from the configured source list (NeetCode 150 / Blind 75 / your own list), capped at N new cards per day.
Why self-grading, not auto-grading Runtime and attempt count are noisy signals (you might've re-read your old solution, or fought a silly typo). Anki's insight is that a one-tap honest self-assessment beats inferred difficulty. Auto-suggesting a grade from solve time is a nice v2 feature — but the human confirms.

How It's Built

Architecture

A Manifest V3 extension with two content scripts, a tiny scheduling library, and chrome.storage.local as the single source of truth. No servers, no accounts — your deck lives in your browser.

Key technical decisions

  • Manifest V3, content-script-only. No background service worker needed for v1 — all logic runs when a LeetCode page is open. Permissions: storage plus host access to leetcode.com.
  • LeetCode is a SPA. The problems table renders late and survives client-side navigation. Use a MutationObserver on document.body to detect the table appearing, and re-check on URL changes (observe history.pushState via a popstate/poll hybrid). Never assume the DOM is ready at document_idle.
  • Detecting "Accepted". Primary: observe the result panel DOM for the accepted state. Fallback/bootstrap: query LeetCode's same-origin GraphQL endpoint (recentAcSubmissionList) from the content script — it rides your existing session cookie, no auth flow needed. This also lets the extension seed the deck from your historical solves on first run.
  • Problem identity = titleSlug. Problem numbers and titles can be parsed from the page, but the slug in the URL (/problems/two-sum/) is the stable key.
  • Resilience to DOM changes. All selectors live in one selectors.js file with a fallback chain, so a LeetCode redesign is a one-file fix. The injected card is built from the extension's own styles (shadow DOM) so LeetCode's CSS can't break it.

Storage Shape

Data model

One record per problem, keyed by slug. Dates are ISO strings; the review log is append-only so stats and future algorithm changes (e.g. migrating to FSRS) can be recomputed from history.

And the entire SM-2 update, for scale — the algorithm is genuinely this small:

// lib/sm2.js — the whole engine
function rate(card, grade) {
  if (grade === "again")
    return { ...card, reps: 0, intervalDays: 1,
             ease: Math.max(1.3, card.ease - 0.2) };

  const ease = grade === "hard" ? Math.max(1.3, card.ease - 0.15)
             : grade === "easy" ? card.ease + 0.15 : card.ease;

  const intervalDays =
      card.reps === 0 ? 1
    : card.reps === 1 ? 6
    : Math.round(card.intervalDays *
        (grade === "hard" ? 1.2 : grade === "easy" ? ease * 1.3 : ease));

  return { ...card, ease, intervalDays, reps: card.reps + 1,
           dueDate: addDays(today(), intervalDays) };
}

Build Order

Roadmap

Four phases, each one shippable. Phase 1 alone is already useful.

i
Goal · a card appears on the page

Skeleton & injection

  • Manifest V3 scaffold; load unpacked in Chrome
  • Content script that reliably finds the problem table (MutationObserver + SPA nav handling)
  • Inject a styled review card (shadow DOM) with a hardcoded problem
  • lib/sm2.js + lib/select.js written and unit-tested with plain Node — before any UI uses them
ii
Goal · the loop closes

Storage & the review loop

  • Deck persistence in chrome.storage.local
  • Problem-page script: detect Accepted, show the rating overlay
  • Rating updates the card via SM-2; the problemset card reflects the new state
  • Manual "add to deck" button on any problem page
iii
Goal · zero manual setup

Seeding & new-card flow

  • First-run import: pull historical accepted submissions via LeetCode GraphQL and seed the deck
  • Bundle NeetCode 150 / Blind 75 as new-card source lists
  • "Nothing due → introduce a new problem" selection, with per-day cap
iv
Goal · it feels like a product

Popup, stats & polish

  • Popup: due queue, streak, deck browser, settings
  • JSON export / import (backup, machine moves)
  • Stats: retention rate, reviews-per-week, ease distribution
  • Stretch: swap SM-2 for FSRS; suggested grade from solve time
07
Known Hazards

Risks & open questions

LeetCode redesigns its DOM

The problem table and result panel have no stable public contract; class names are hashed and change.

Mitigation — all selectors centralized with fallback chains; prefer structural anchors (the link grid under /problemset/) over class names; GraphQL fallback for solve detection.

SPA timing flakiness

Injecting too early gets wiped by React hydration; too late and the card pops in jarringly.

Mitigation — observe for the table, inject once it's stable for ~100ms, and re-assert presence on mutations (idempotent inject keyed by element id).

Single-device deck

chrome.storage.local doesn't follow you across machines.

Mitigation — v1: JSON export/import. Later: chrome.storage.sync (size limits apply — review log may need pruning) or rebuild-from-GraphQL on a new machine.

Grading honesty

Self-rating only works if you're honest; "Easy" inflation collapses the schedule's value.

Mitigation — show solve-time context next to the buttons ("you took 34 min — sure this was Easy?") as a gentle nudge, not an override.
